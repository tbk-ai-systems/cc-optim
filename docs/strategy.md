# cc-optim: Optimization Strategy

## What is this

A CLI tool (`npx cc-optim`) that sets up an optimized Claude Code environment in any project. It eliminates the "Context Tax" - the wasted tokens and time Claude Code spends re-reading and exploring files it doesn't need to.

The system consists of **three layers** that give Claude pre-built context instead of letting it explore blindly.

A future version will include a transparent **Architect-Constructor pipeline** (Opus plans, local LLM executes) implemented as a Claude Code skill/hook.

---

## The Three Layers

### Layer 1: Structural (The Code Map)

**Problem it solves**: Claude Code wastes thousands of tokens opening and reading files one by one to understand your project structure.

**Solution**: Package the entire codebase into a single structured XML file using **Repomix**. Claude reads this once (cached by Anthropic's prompt caching), then accesses any part instantly.

**What gets installed/generated**:

| What | Where | How |
|------|-------|-----|
| Repomix CLI | devDependency (`npm install --save-dev repomix`) | cc-optim installs it (uses detected pkg manager) |
| Repomix MCP server | Claude Code config (`claude mcp add repomix -- npx -y repomix --mcp`) | cc-optim configures it |
| `.repomixignore` | Project root | cc-optim generates from template (like .gitignore, excludes node_modules, dist, images, etc.) |
| `repomix.config.json` | Project root | cc-optim generates (output format, security checks, compression settings) |
| `.claudeignore` | Project root | cc-optim generates (prevents Claude from reading raw files it doesn't need - build artifacts, lock files, etc.) |

**How it works at runtime**:
1. User runs `repomix` (or it runs via git hook) → produces `repomix-output.xml`
2. Claude Code reads `repomix-output.xml` via MCP server or directly
3. Anthropic's prompt caching stores it → subsequent reads are near-free
4. `.claudeignore` blocks Claude from wastefully reading raw source files that are already in the XML

**When to regenerate**: After significant code changes. Configurable via git hook (auto or reminder-only).

---

### Layer 2: Semantic (The Rules & Plan)

**Problem it solves**: Even with the full codebase, Claude doesn't know *why* you built things a certain way, what the conventions are, or what you're working on next. It guesses, and guesses wrong.

**Solution**: Living documentation files that Claude reads **before** touching any code. These files are the "brain" that tells Claude the rules of your project.

**What gets generated**:

| What | Where | Purpose |
|------|-------|---------|
| `CLAUDE.md` | Project root | **The soul**. Claude Code reads this automatically on every session start. Contains: project overview, strict rules ("read ARCHITECTURE.md first"), commit rules ("update plan-registry before committing"), test/lint commands, pointers to all other docs |
| `ARCHITECTURE.md` | `docs/ARCHITECTURE.md` | Architecture overview: tech stack, directory structure, data flow, design patterns, conventions. Either a pre-filled skeleton with `<!-- TODO -->` markers, OR auto-generated by Claude Code using the Repomix XML (user chooses in TUI). If existing ARCHITECTURE.md found, it's passed as merge input. |
| `spec-registry.md` | `docs/spec-registry.md` | **Feature index**. A table listing every feature/epic with status and link to its detailed plan file. This is the "table of contents" |
| `plan-registry.md` | `docs/plan-registry.md` | **Active execution checklist**. Step-by-step tasks with `[ ]` / `[x]` checkboxes. What the AI (or developer) is currently working through |
| `docs/plans/` | Directory | **Multi-planning folder**. One markdown file per feature/epic (e.g., `auth-plan.md`, `payments-plan.md`). Prevents monolithic plans from polluting context. **Cleanup policy**: Keep `[x]` tasks while the feature is in progress; delete the plan file once the feature is fully shipped. |
| `docs/plans/_template.md` | Template file | **Plan format template**. Shows the expected structure for feature plans: Objective, Steps with checkboxes, Acceptance Criteria, Notes. Claude uses this format when creating new plans. |

**How it works at runtime**:
1. Claude Code starts → reads `CLAUDE.md` (automatic, native behavior)
2. `CLAUDE.md` says: "Before doing anything, read `docs/ARCHITECTURE.md`"
3. `CLAUDE.md` says: "To find what to work on, read `docs/spec-registry.md`"
4. `CLAUDE.md` says: "Before committing, update `docs/plan-registry.md`"
5. Claude follows these rules throughout the session

**Maintenance**: The `CLAUDE.md` rules instruct Claude to update these files as part of its workflow. The developer validates changes during code review.

### Layer 3: Memory (Persistent Context)

**Problem it solves**: When you close your terminal and reopen Claude Code, it forgets everything from the previous session. You waste time re-explaining context.

**Solution**: Two mechanisms working together:
- **Grov**: A proxy that intercepts Claude Code sessions and stores conversational reasoning in a local SQLite database. On next session, it injects the relevant context automatically.
- **Git hooks**: Ensure documentation stays up-to-date, so even without Grov, Claude can reconstruct context from the semantic layer files.

**What gets installed/configured**:

| What | Where | How |
|------|-------|-----|
| Grov CLI | devDependency (`npm install --save-dev grov`) | cc-optim installs it (uses detected pkg manager) |
| Grov config | `~/.grov/` (user home) | `grov init` creates it |
| Grov memory DB | `~/.grov/memory.db` (SQLite) | Created automatically by grov |
| Git hooks dir | `.githooks/` in project | cc-optim creates the directory |
| Pre-commit hook | `.githooks/pre-commit` | cc-optim generates (configurable: auto/reminder/none). Auto mode checks for substantial source changes before running repomix. chmod 755 applied. Uses `#!/usr/bin/env bash` shebang for Windows Git Bash compatibility. |
| Git hooks path | `.git/config` (`core.hooksPath`) | cc-optim runs `git config core.hooksPath .githooks` |

**How it works at runtime**:
1. Developer starts session: `grov claude` (instead of just `claude`)
2. Grov injects relevant past context into the session
3. Developer works normally with Claude Code
4. On `git commit`: pre-commit hook fires
   - **Auto mode**: Uses a stash trick to safely regenerate repomix mid-commit. Checks if staged files include substantial source changes (ignoring docs/, tests/, configs, .md, .json, .lock). If real source changed: creates a temp commit, stashes unstaged changes, resets the temp commit, runs `repomix`, stages the output, then pops the stash. This avoids mixing unstaged work into the repomix snapshot.
     ```bash
     #!/usr/bin/env bash
     SOURCE_CHANGES=$(git diff --cached --name-only | grep -vE '^(docs/|tests?/|__tests__/|\..*ignore|.*\.md|.*\.json|.*\.lock)$')
     if [ -n "$SOURCE_CHANGES" ]; then
       echo "cc-optim: Source files changed, regenerating repomix..."
       git commit -m "temp" --no-verify > /dev/null 2>&1
       HAS_UNSTAGED=false
       if ! git diff-index --quiet HEAD --; then
           HAS_UNSTAGED=true
           git stash push -k -q -m "cc-optim-temp-stash"
       fi
       git reset --soft HEAD~1 > /dev/null 2>&1
       npx repomix
       git add repomix-output.xml 2>/dev/null || git add repomix-output.md 2>/dev/null
       echo "cc-optim: repomix output updated and staged"
       if [ "$HAS_UNSTAGED" = true ]; then
           git stash pop -q
       fi
     fi
     ```
   - **Reminder mode**: Echoes "Remember to run repomix and update ARCHITECTURE.md if structure changed"
   - **None**: No hook installed
5. Grov stores the session reasoning in SQLite for next time

**Grov architecture**:
```
Terminal (claude) → Grov Proxy (localhost) → Anthropic API
                         ↓
                    SQLite (~/.grov/memory.db)
                         ↓
                    Next session: injects stored context
```

## The Architect-Constructor Pipeline (Future Version)

**Concept**: Use an expensive high-reasoning model (Claude Opus) to design architecture and plans, then use a cheaper/local model to execute the code step-by-step.

**NOT in v1**. Will be implemented as a transparent Claude Code skill/hook in a future version:
1. After user approves a plan in Claude Code, the skill asks: "Execute with local model?"
2. If yes: creates a temporary commit (clean slate) → local LLM executes the plan steps → Claude Code reviews the git diff
3. If execution looks correct: squash temp commit + changes into a proper commit
4. If execution failed or has too many issues: ask user if they want to regenerate the plan with Claude normally

This is transparent to the user — no manual Aider/Ollama setup needed.

## How the CLAUDE.md "soul" works in practice

The most important generated file. Here's what it tells Claude to do:

```markdown
# Project: {name}
{description}

## Rules

### Before Starting Work
1. Read `docs/ARCHITECTURE.md` to understand the system architecture
2. Read `docs/spec-registry.md` to see the feature roadmap
3. Read `docs/plan-registry.md` to see current active tasks
4. Do NOT explore the codebase blindly - use the docs first

### During Work
- Follow the conventions in ARCHITECTURE.md
- For new features: create a plan in `docs/plans/{feature}-plan.md` first
- Update `docs/spec-registry.md` when adding new features
- Use Plan Mode (Shift+Tab) for complex changes
- **Multi-plan support**: Multiple plans can be active simultaneously. Check `docs/plan-registry.md` to see which plans are in progress. Work on the plan the user specifies, or ask which one if ambiguous.
- **Proactive questioning**: When a task is ambiguous, under-specified, or has multiple valid approaches, ASK the user before proceeding. Do not guess. Prefer a 1-sentence clarifying question over 100 lines of wrong code. Examples:
  - "Should this be a new file or added to the existing module?"
  - "The plan says X but the code already does Y — which takes priority?"
  - "This change would affect Z — should I proceed or scope it down?"

### Before Committing
- Update `docs/plan-registry.md` (mark completed steps, add new ones)
- If architecture changed: update `docs/ARCHITECTURE.md`
- If new feature added: update `docs/spec-registry.md`
- Run: {test command}
- Run: {lint command}

### Context Sources
- Full codebase snapshot: `repomix-output.xml` (use Repomix MCP server)
- Architecture & conventions: `docs/ARCHITECTURE.md`
- Feature roadmap: `docs/spec-registry.md`
- Current tasks: `docs/plan-registry.md`
```

## Idempotency

Running `npx cc-optim` multiple times is safe:
- If a file already exists: asks "Overwrite? / Skip? / Merge?"
- If a tool is already installed: skips installation
- If git hooks are already set up: skips hook setup
- New runs can add layers that were skipped before
